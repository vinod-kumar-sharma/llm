## 🏗️ CATEGORY 2: ARCHITECTURE & DESIGN

**Focus:** Structural patterns, design principles, integration styles, and system quality attributes.

| Word | Meaning | Hindi | Example |
|------|---------|-------|---------|
| **Abstraction** | Hiding complex implementation details | अमूर्तता | "The API layer provides an abstraction over the DB." |
| **Encapsulation** | Bundling data and methods together | encapsulation (संपुटीकरण) | "Encapsulation prevents direct access to internal state." |
| **Polymorphism** | Ability to present the same interface for different forms | बहुरूपता | "Polymorphism allows different shapes to implement 'draw'." |
| **Inheritance** | Mechanism where a new class derives from an existing class | विरासत | "The 'Car' class uses inheritance from 'Vehicle'." |
| **Coupling** | Degree of interdependence between modules | युग्मन | "Tight coupling makes the system hard to test." |
| **Cohesion** | Degree to which elements of a module belong together | सामंजस्य | "High cohesion usually leads to low coupling." |
| **Single Responsibility Principle** | A class should have one and only one reason to change | एकल जिम्मेदारी सिद्धांत | "This class violates SRP by handling logging and logic." |
| **Open/Closed Principle** | Open for extension, closed for modification | खुला/बंद सिद्धांत | "Use plugins to adhere to the Open/Closed Principle." |
| **Liskov Substitution** | Subtypes must be substitutable for their base types | लिस्कोव प्रतिस्थापन | "The derived class fails Liskov Substitution checks." |
| **Interface Segregation** | Clients should not depend on interfaces they don't use | इंटरफ़ेस पृथक्करण | "Split the huge interface to follow Interface Segregation." |
| **Dependency Inversion** | Depend on abstractions, not concretions | निर्भरता व्युत्क्रमण | "Use Dependency Injection to satisfy Dependency Inversion." |
| **Design Pattern** | Reusable solution to a common problem | डिजाइन पैटर्न | "Singleton is a common design pattern." |
| **Anti-Pattern** | Common response to a recurring problem that is usually ineffective | एंटी-पैटर्न | "God Object is a dangerous anti-pattern." |
| **Big Ball of Mud** | System with no discernible architecture | बिग बॉल ऑफ मड | "The legacy code has become a Big Ball of Mud." |
| **Singleton** | Class with only one instance | सिंगलटन | "The database connection pool uses a Singleton." |
| **Factory Method** | Interface for creating objects in a superclass | फैक्ट्री विधि | "Use a Factory to instantiate the payment providers." |
| **Adapter Pattern** | Allows incompatible interfaces to work together | एडाप्टर पैटर्न | "The Adapter wraps the old XML service for JSON consumers." |
| **Facade Pattern** | Simplified interface to a complex subsystem | अग्रभाग (Facade) पैटर्न | "The Facade hides the complexity of the billing system." |
| **Observer Pattern** | Subscription mechanism to notify multiple objects | ऑब्जर्वर पैटर्न | "The UI updates using the Observer pattern." |
| **Strategy Pattern** | Interchangeable algorithms | रणनीति पैटर्न | "Switch sorting algorithms using the Strategy pattern." |
| **Decorator Pattern** | Dynamically adding behavior to an object | डेकोरेटर पैटर्न | "Use Decorators to add logging to the streams." |
| **Proxy Pattern** | Placeholder for another object to control access | प्रॉक्सी पैटर्न | "The Proxy handles lazy loading of large images." |
| **Event-Driven Architecture** | Design promoting the production/consumption of events | घटना-संचालित वास्तुकला | "EDA decouples producers from consumers." |
| **Microkernel Architecture** | Core system with plug-in modules | माइक्रोकर्नेल वास्तुकला | "Eclipse uses a Microkernel architecture." |
| **Space-Based Architecture** | Tuple-space concepts for scaling | अंतरिक्ष-आधारित वास्तुकला | "Space-based architecture removes the central database bottleneck." |
| **Serverless Architecture** | Relying on third-party services (BaaS) or FaaS | सर्वररहित वास्तुकला | "Serverless architecture reduces our ops overhead." |
| **Hexagonal Architecture** | Ports and Adapters architecture | हेक्सागोनल वास्तुकला | "Hexagonal architecture isolates the core domain logic." |
| **CQRS** | Command Query Responsibility Segregation | CQRS | "CQRS allows us to scale reads independently of writes." |
| **Event Sourcing** | Storing state as a sequence of events | इवेंट सोर्सिंग | "Event sourcing gives us a perfect audit trail." |
| **Idempotency** | Making multiple identical requests has the same effect | कामचलाऊ (Idempotency) | "Payment APIs must guarantee idempotency." |
| **Stateless** | No session information retained by receiver | स्टेटलेस | "REST APIs should be stateless for scalability." |
| **Stateful** | Receiver retains session information | स्टेटफुल | "WebSockets are a stateful connection." |
| **Orchestration** | Central controller manages interaction | आर्केस्ट्रा (Orchestration) | "The saga orchestrator manages the transaction workflow." |
| **Choreography** | Each component knows its own role (decentralized) | कोरियोग्राफ़ी | "Event choreography avoids a central bottleneck." |
| **Vertical Scaling** | Adding more power to an existing machine (Scale Up) | वर्टिकल स्केलिंग | "We hit the limit of vertical scaling on the DB." |
| **Horizontal Scaling** | Adding more machines to the pool (Scale Out) | हॉरिजॉन्टल स्केलिंग | "Horizontal scaling is cheaper than buying a mainframe." |
| **Load Balancing** | Distributing network traffic across servers | लोड संतुलन | "The load balancer handles the traffic spike." |
| **Caching** | Storing copies of data for faster access | कैशिंग | "Redis caching reduced DB load by 90%." |
| **Sharding** | Partitioning data across multiple databases | शार्डिंग | "We actived sharding based on User ID." |
| **Replication** | Copying data to multiple locations | प्रतिकृति | "Read replication improves query performance." |
| **Consistency** | Data is the same across all nodes | स्थिरता (Consistency) | "Strong consistency is required for financial transactions." |
| **Availability** | System is operational and accessible | उपलब्धता | "We trade consistency for availability in this AP system." |
| **Partition Tolerance** | System continues despite network failures | विभाजन सहिष्णुता | "Cassandra is designed for Partition Tolerance." |
| **CAP Theorem** | Pick two: Consistency, Availability, Partition Tolerance | CAP प्रमेय | "Understand the CAP theorem trade-offs." |
| **BASE** | Basically Available, Soft state, Eventual consistency | BASE मॉडल | "NoSQL databases often follow the BASE model." |
| **ACID** | Atomicity, Consistency, Isolation, Durability | ACID मॉडल | "Relational DBs guarantee ACID transactions." |
| **Throughput** | Rate of successful message delivery | थ्रूपुट | "We need a throughput of 10k requests per second." |
| **Latency** | Time from request to response | विलंबता | "Network latency is causing the UI lag." |

## 🛡️ CATEGORY 4: RELIABILITY, PR & DR

**Focus:** System stability, chaos engineering, incident response, and recovery metrics.

| Word | Meaning | Hindi | Example |
|------|---------|-------|---------|
| **High Availability (HA)** | System remains operational for long periods | उच्च उपलब्धता | "HA requires redundant components." |
| **Fault Tolerance** | System continues operating properly in event of failure | दोष सहिष्णुता | "Fault tolerance is built into the disk array." |
| **Redundancy** | Duplication of critical components | अतिरेक (Redundancy) | "Power redundancy prevented a shutdown." |
| **Single Point of Failure (SPOF)** | Part that stops entire system if it fails | विफलता का एकल बिंदु | "The load balancer is a SPOF." |
| **SLA (Service Level Agreement)** | Commitment to service quality (Contract) | SLA | "We owe credits if we breach the SLA." |
| **SLO (Service Level Objective)** | Target value for a service level (Internal) | SLO | "Our internal SLO is stricter than the client SLA." |
| **SLI (Service Level Indicator)** | Measure of the service level (Metric) | SLI | "Latency is the primary SLI we track." |
| **Error Budget** | Allowed amount of unreliability | त्रुटि बजट | "We exhausted our error budget for the month." |
| **MTTR (Mean Time To Recovery)** | Avg time to fix a failure | वसूली का औसत समय | "Reducing MTTR is our top priority." |
| **MTBF (Mean Time Between Failures)** | Avg time between breakdowns | विफलताओं के बीच का औसत समय | "Hardware MTBF is usually years." |
| **RTO (Recovery Time Objective)** | Max acceptable downtime duration | RTO | "The RTO for the payment gateway is 5 minutes." |
| **RPO (Recovery Point Objective)** | Max acceptable data loss | RPO | "Daily backups give us a 24-hour RPO." |
| **Disaster Recovery (DR)** | Procedures to recover from major disruption | आपदा वसूली | "Initiate the DR plan immediately." |
| **Business Continuity Plan (BCP)** | Process to continue operations during disaster | BCP | "The BCP includes remote work protocols." |
| **Failover** | Switching to a redundant system | फेलओवर | "Automatic failover to the secondary DB succeeded." |
| **Failback** | Returning to the primary system | फेलबैक | "Plan the failback during the maintenance window." |
| **Active-Active** | Two systems handling load simultaneously | सक्रिय-सक्रिय | "Active-Active clusters provide load balancing and DR." |
| **Active-Passive** | One system handles load, other waits | सक्रिय-निष्क्रिय | "Active-Passive is cheaper but failover is slower." |
| **Warm Standby** | Backup system running but not full scale | वार्म स्टैंडबाई | "The DR site is in warm standby mode." |
| **Pilot Light** | Minimal version of environment running | पायलट लाइट | "Pilot light DR saves costs until needed." |
| **Cold Standby** | Backup system turned off until needed | कोल्ड स्टैंडबाई | "Cold standby has the longest RTO." |
| **Chaos Engineering** | Experimenting to find system weaknesses | अराजकता इंजीनियरिंग | "Chaos engineering helped us find the race condition." |
| **Chaos Monkey** | Tool that randomly terminates instances | कैओस मंकी | "Chaos Monkey runs in production during business hours." |
| **Blast Radius** | Impact area of a failure | विस्फोट त्रिज्या | "Limit the blast radius by using cell architecture." |
| **Game Day** | Dedicated time to practice incident response | गेम डे | "The SRE team is hosting a Game Day on Friday." |
| **Incident Commander** | Person in charge during an incident | घटना कमांडर | "The Incident Commander coordinates the response." |
